\subsection{Detailed Description}
\doxyref{Throwable}{p.}{classjava_1_1lang_1_1_throwable} is the superclass of all exceptions that can be raised.

There are two special cases: \doxyref{Error}{p.}{classjava_1_1lang_1_1_error} and \doxyref{RuntimeException}{p.}{classjava_1_1lang_1_1_runtime_exception}: these two classes (and their subclasses) are considered unchecked exceptions, and are either frequent enough or catastrophic enough that you do not need to declare them in {\ttfamily throws} clauses. Everything else is a checked exception, and is ususally a subclass of \doxyref{Exception}{p.}{classjava_1_1lang_1_1_exception}; these exceptions have to be handled or declared.

Instances of this class are usually created with knowledge of the execution context, so that you can get a stack trace of the problem spot in the code. Also, since JDK 1.4, Throwables participate in \char`\"{}exception
 chaining.\char`\"{} This means that one exception can be caused by another, and preserve the information of the original.

One reason this is useful is to wrap exceptions to conform to an interface. For example, it would be bad design to require all levels of a program interface to be aware of the low-\/level exceptions thrown at one level of abstraction. Another example is wrapping a checked exception in an unchecked one, to communicate that failure occured while still obeying the method throws clause of a superclass.

A cause is assigned in one of two ways; but can only be assigned once in the lifetime of the \doxyref{Throwable}{p.}{classjava_1_1lang_1_1_throwable}. There are new constructors added to several classes in the exception hierarchy that directly initialize the cause, or you can use the {\ttfamily initCause} method. This second method is especially useful if the superclass has not been retrofitted with new constructors:\par
 
\begin{DoxyPre}
 try
   \{
     lowLevelOp();
   \}
 catch (LowLevelException lle)
   \{
     throw (HighLevelException) new HighLevelException().initCause(lle);
   \}
 \end{DoxyPre}
 Notice the cast in the above example; without it, your method would need a throws clase that declared \doxyref{Throwable}{p.}{classjava_1_1lang_1_1_throwable}, defeating the purpose of chainig your exceptions.

By convention, exception classes have two constructors: one with no arguments, and one that takes a \doxyref{String}{p.}{classjava_1_1lang_1_1_string} for a detail message. Further, classes which are likely to be used in an exception chain also provide a constructor that takes a \doxyref{Throwable}{p.}{classjava_1_1lang_1_1_throwable}, with or without a detail message string.

Another 1.4 feature is the StackTrace, a means of reflection that allows the program to inspect the context of the exception, and which is serialized, so that remote procedure calls can correctly pass exceptions.
